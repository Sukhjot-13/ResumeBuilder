# Audit Report — Resume Builder Codebase
> Generated: 2026-05-24 | Last updated: 2026-05-24

## Framework & Tech Stack
- **Framework**: Next.js 16.0.7 (App Router), React 19.2.0
- **Auth**: Custom JWT-based auth (JOSE for Edge middleware, jsonwebtoken for Node) with OTP email login (Brevo)
- **Database**: MongoDB via Mongoose (8.19.3)
- **AI**: Gemini Flash (`gemini-flash-latest`) via unified AI config at `src/lib/ai/`
- **PDF**: `@react-pdf/renderer` v4.3.1
- **Styling**: Tailwind CSS v4
- **UI Components**: Custom (no shadcn/ui)

## Auth
- Custom JWT auth with OTP-based email login (Brevo)
- Access tokens (15min) + refresh tokens (15 days) with rotation
- Middleware proxy (`src/proxy.js`) reads cookies → verifies tokens → injects `x-user-id` header
- Auth context (`AuthContext.js`) on client side calls `/api/user/profile` to check auth state
- Role-based permission system (ADMIN=0, DEVELOPER=70, SUBSCRIBER=99, USER=100)
- Credit tracking (Free: 2/day, Pro: 200/month)

## Database
- **MongoDB** (not Supabase/Postgres)
- **5 models**: User, Resume, ResumeMetadata, RefreshToken, Transaction

## Resume Generation
- **Flow**: API route → `resume-generator.js` → `callAI('RESUME_GENERATION')` → Gemini Flash
- **Prompt strategy**: `src/lib/promptConfig.js` (3 tiers: basic/standard/premium per role)
- **AI Edit**: `aiResumeEditorService.js` → `callAI('AI_EDIT')` → Gemini Flash
- **Resume Parsing**: `resumeParsingService.js` → `callAI('RESUME_PARSING')` → Gemini Flash
- **Gatekeeper**: `callAI('GATEKEEPER')` → Gemini Flash (evaluates job listings for automation)
- **Schema source of truth**: `src/lib/resumeFields.js` → auto-generates Mongoose schemas, AI prompts, and form state

## PDF Generation
- **Library**: `@react-pdf/renderer` (v4.3.1)
- **Flow**: API route → `pdf-generator.js` → react-pdf → Buffer
- **Templates** (6): ClassicTemplate, ClassicTemplate2, Creative, Modern, Professional, Simple

### Dashboard Placement
- **Same Next.js app at `/automation/*`** — not a separate deploy. Reuses existing custom JWT auth, MongoDB connection, and deployment.

  - Copy `/automation/*` pages + API routes → new Next.js app
  - Copy Mongoose models + DB connection utility → both apps
  - Copy auth utilities (`AuthContext`, `requirePermission`, `jwt.js`) → new app
  - Add `MONGODB_URI` + `JWT_SECRET` to new app's env vars (same MongoDB, same JWT secret)
  - CORS headers only needed on Resume Builder routes called by the new Dashboard
  - No auth rewrite — stateless JWT works on both apps with the same secret

### Gatekeeper AI
- **Gemini Flash initially** via existing `callAI('GATEKEEPER')` in `src/lib/ai/config.js`. Worker calls it through `POST /api/gatekeeper/evaluate` API route (HTTP), not directly.

- **API Keys** — `ApiKey` Mongoose model + Bearer token middleware. Future-proof for when they're separate services.

### Templates

## Session 3 — Code Review Findings

> Generated: 2026-05-24 | Automation code full review

### Critical Bugs

**1. `generate-content` route doesn't return `resumeId`**
- `worker/src/queue/processors/generate.processor.js:18` does `result.resumeId` but `POST /api/generate-content` returns `ok(tailoredData)` — the AI content, not a document with `resumeId`
- Breaks the generate → apply handoff: `resumeId` is always `undefined`

**2. `generateResume` called with empty user profile in worker**
- `worker/src/db-api.js:49-50`: `generateResume(jobDescription, userProfile)` → called as `generateResume(desc, {})`
- Route expects `{ resume, jobDescription }` but worker passes `{ jobDescription, userProfile }`

### Important Issues

**3. Gatekeeper doesn't persist AI decisions to DB**
- `src/app/api/gatekeeper/evaluate/route.js` returns decision to caller but never saves to `GatekeeperDecision` model
- No audit trail, no ability to review past decisions

**4. `GatekeeperRules` null breaks the AI prompt**
- `src/app/api/gatekeeper/evaluate/route.js:51`: if user hasn't saved rules, `rules` is `null`
- Prompt becomes `USER RULES:\nnull` followed by job — AI may misinterpret

**5. `createBrowserContext` returns `context` but callers leak the browser process**
- `worker/src/automation/browser.js:35` returns `context`, not `browser`
- Callers call `browser.close()` which is actually `context.close()` — browser process stays running
- Not causing test failure but wastes resources

**6. AI test stuck on "Continue" button — iframe detection fix applied**
- `worker/src/test-apply-with-ai.js`: Indeed opens an iframe for the apply modal after clicking "Apply with Indeed"
- Original code only checked for iframes once BEFORE the AI loop → `target` stayed on main page
- `getContent()` read main page which never changed → stuck detection after 3 identical cycles
- Also: `setTimeout(() => b.click(), 100)` inside `page.evaluate()` fires asynchronously after evaluate returns — click often missed
- **Fix applied batch 1 (may need more refinement)**:
  - `detectTarget()` called after every action; `executeActions` uses Playwright locator API instead of evaluate+setTimeout
  - Text-based clicks now use `target.locator().filter({ hasText })` — works on both Page and Frame
  - Frame detection widened: picks first non-blank, non-main frame instead of matching specific URL patterns (Indeed changes iframe URLs per form step)

**7. Gate processor fetches ALL pending jobs just to find one**
- `worker/src/queue/processors/gate.processor.js:11-12`: `getPendingJobListings()` returns unlimited results, then `.find()` in JS
- Should use a dedicated `getJobListingById(id)` instead

**8. Scrape processor also fetches ALL pending jobs**
- Same issue: `getPendingJobListings()` in `worker/src/queue/processors/scrape.processor.js:39` with no limit
- Could return hundreds/thousands of jobs, all enqueued to gate queue

**9. API key GET query doesn't check `isActive`**
- `src/app/api/api-keys/route.js:15`: queries `revokedAt: null` but misses `isActive: true`
- A key with `isActive: false` but no `revokedAt` date would still appear in the list

**10. Sessions GET exposes encrypted cookies in response**
- `src/app/api/automation/sessions/route.js:17`: `.select(... 'cookiesEncrypted' ...)` — includes encrypted cookies in listings
- While encrypted, still sensitive — should be excluded from GET response

**11. Proxy passes all Bearer tokens through without validation**
- `src/proxy.js:27-30`: any request with `Authorization: Bearer <anything>` is passed through
- Intentional (DB lookup needed to validate), but invalid API keys still reach route handlers

### Medium Issues

**12. Empty directories**
- `worker/src/gatekeeper/` and `worker/src/resume-client/` — empty. Dead scaffolding or unfinished work.

**13. External ID dedup is weak**
- `worker/src/scraper/linkedin.js:49`, `indeed.js:49`: `externalId = \`${company}-${title}\`.replace(...)` — same company+title in different cities collides

**14. Job detail page linked but doesn't exist**
- `src/app/automation/jobs/page.js:111`: links to `/automation/jobs/${job._id}` but no `[id]/page.js` route exists

**15. `GatekeeperRules` and `JobCriteria` duplicate fields**
- Both have `targetTitles`/`titles`, `minSalary`, work mode booleans, seniority levels
- Unclear which is authoritative for each concern

**16. No rate limiting on API key routes**
- No rate limiting on generate-content or other API-key-authenticated endpoints

### Minor Issues

**17. `render.yaml` references `npm run build`** but `package.json` has no build script

**18. `scrapeIntervalMs` in worker config is unused** — scheduler runs on hardcoded `*/30 * * * *` cron

**19. `COOKIE_ENCRYPTION_KEY` env var** checked in both Next.js app (`src/config/env.js`) and worker (`worker/src/config.js`) with different fallback behavior — Next.js doesn't validate it's set

### Verified Working (No Issues)
- **Indeed cookies**: Cookie injection logic identical between `test-apply-with-ai.js` and `apply.processor.js` (same decrypt flow, same `addCookies()` call, same domain/path/sameSite mapping)
- **API key auth middleware**: `authenticateRequest()` in `apiKeyAuth.js` correctly hashes + compares API keys via SHA-256
- **Encryption**: `encrypt()`/`decrypt()` in `encryption.js` uses AES-256-GCM correctly
- **All model schemas**: Indexes correct, validation present, no missing fields
- **All dashboard pages**: Loading/empty/error states handled, permission-gated at layout level
- **Scheduler cron**: Properly checks time window, enabled flag, and active days before dispatching

## Session 4 — AI Apply Test Fixes

> Date: 2026-05-24 | Fixes applied to `test-apply-with-ai.js` and `apply.processor.js`

### Problems Found & Fixed

**1. Iframe detection ran once before the AI loop**
- `detectTarget()` now runs after every action, prefers `smartapply`/`indeedapply` frames, and **excludes reCAPTCHA/hCAPTCHA frames**
- If main page URL is an apply form page (contains `indeedapply`), uses main page directly

**2. Text-based clicks used evaluate + setTimeout (unreliable)**
- Replaced with Playwright locators: `target.locator().filter({ hasText })` — works on both Page and Frame
- Added fallback chain: `scrollIntoViewIfNeeded` (2s timeout) → `click()` (5s timeout) → `el.evaluate(el => el.click())`
- Section 6 apply button click uses `getByRole()` / `getByText()` instead of `setTimeout` inside `page.evaluate()`

**3. No wait for Smart Apply iframe**
- Added `waitForApplyFrame()` — polls every 1s for 15s waiting for the `smartapply.indeed.com` frame
- Added `detectTarget()` which filters out CAPTCHA frames and prioritizes Indeed apply frames

**4. getContent() missed Shadow DOM elements**
- Added recursive shadow root traversal via `collect()` function
- Added `waitForSelector` (5s timeout) before evaluate to let React content render
- Added ARIA role selectors (`[role="button"]`, `[role="textbox"]`, `[role="combobox"]`)

**5. CAPTCHA detection was hard exit**
- Now: wait 4s → re-check content → if still blocked, wait 5s + 5s for manual resolution
- After wait, re-navigates to original job URL if page was redirected away (but skips re-nav if already on an apply form page)

**6. apply.processor.js ported improvements**
- Added `detectTarget()` and `waitForApplyFrame()` for iframe handling
- Added `resilientClick()` with force-click fallback
- Added CAPTCHA detection with 10s manual wait loop
- Added form step navigation (tries Continue/Next/Submit buttons in frame)
