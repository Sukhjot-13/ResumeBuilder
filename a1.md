# Audit Report — Resume Builder Codebase
> Generated: 2026-05-24 | Last updated: 2026-05-24

## Framework & Tech Stack
- **Framework**: Next.js 16.0.7 (App Router), React 19.2.0
- **Auth**: Custom JWT-based auth (JOSE for Edge middleware, jsonwebtoken for Node) with OTP email login (Brevo)
- **Database**: MongoDB via Mongoose (8.19.3)
- **AI**: Gemini Flash (`gemini-flash-latest`) via unified AI config at `src/lib/ai/`
- **PDF**: `@react-pdf/renderer` v4.3.1
- **Styling**: Tailwind CSS v4
- **UI Components**: Custom (no shadcn/ui)

## Auth
- Custom JWT auth with OTP-based email login (Brevo)
- Access tokens (15min) + refresh tokens (15 days) with rotation
- Middleware proxy (`src/proxy.js`) reads cookies → verifies tokens → injects `x-user-id` header
- Auth context (`AuthContext.js`) on client side calls `/api/user/profile` to check auth state
- Role-based permission system (ADMIN=0, DEVELOPER=70, SUBSCRIBER=99, USER=100)
- Credit tracking (Free: 2/day, Pro: 200/month)

## Database
- **MongoDB** (not Supabase/Postgres)
- **15 models**: User, Resume, ResumeMetadata, RefreshToken, Transaction, ApiKey, Application, DailyCount, GatekeeperDecision, GatekeeperRules, JobCriteria, JobListing, PlatformSession, SchedulerSettings, plan

## Resume Generation
- **Flow**: API route → `resume-generator.js` → `callAI('RESUME_GENERATION')` → Gemini Flash
- **Prompt strategy**: `src/lib/promptConfig.js` (3 tiers: basic/standard/premium per role)
- **AI Edit**: `aiResumeEditorService.js` → `callAI('AI_EDIT')` → Gemini Flash
- **Resume Parsing**: `resumeParsingService.js` → `callAI('RESUME_PARSING')` → Gemini Flash
- **Gatekeeper**: `callAI('GATEKEEPER')` → Gemini Flash (evaluates job listings for automation)
- **Schema source of truth**: `src/lib/resumeFields.js` → auto-generates Mongoose schemas, AI prompts, and form state

## PDF Generation
- **Library**: `@react-pdf/renderer` (v4.3.1)
- **Flow**: API route → `pdf-generator.js` → react-pdf → Buffer
- **Templates** (6): ClassicTemplate, ClassicTemplate2, Creative, Modern, Professional, Simple

## Worker Code Review Issues (Unresolved)

### Critical Bugs

**1. `generate-content` route doesn't return `resumeId`**
- `worker/src/queue/processors/generate.processor.js:18`: `result.resumeId` but `POST /api/generate-content` returns `ok(tailoredData)` — the AI content, not a document with `resumeId`
- Breaks the generate → apply handoff: `resumeId` is always `undefined`

**2. `generateResume` called with empty user profile in worker**
- `worker/src/db-api.js:49-50`: `generateResume(jobDescription, userProfile)` → called as `generateResume(desc, {})`
- Route expects `{ resume, jobDescription }` but worker passes `{ jobDescription, userProfile }`

**3. Gatekeeper doesn't persist AI decisions to DB**
- `src/app/api/gatekeeper/evaluate/route.js` returns decision to caller but never saves to `GatekeeperDecision` model
- No audit trail, no ability to review past decisions

**4. `GatekeeperRules` null breaks the AI prompt**
- `src/app/api/gatekeeper/evaluate/route.js:51`: if user hasn't saved rules, `rules` is `null`
- Prompt becomes `USER RULES:\nnull` followed by job — AI may misinterpret

**5. `createBrowserContext` returns `context` but callers leak the browser process**
- `worker/src/automation/browser.js:35` returns `context`, not `browser`
- Callers call `browser.close()` which is actually `context.close()` — browser process stays running

### Medium Issues

**6. Gate processor fetches ALL pending jobs just to find one**
- `worker/src/queue/processors/gate.processor.js:11-12`: `getPendingJobListings()` returns unlimited results, then `.find()` in JS
- Should use a dedicated `getJobListingById(id)` instead

**7. Scrape processor also fetches ALL pending jobs**
- Same issue: `getPendingJobListings()` in `worker/src/queue/processors/scrape.processor.js:39` with no limit

**8. API key GET query doesn't check `isActive`**
- `src/app/api/api-keys/route.js:15`: queries `revokedAt: null` but misses `isActive: true`

**9. Sessions GET exposes encrypted cookies in response**
- `src/app/api/automation/sessions/route.js:17`: `.select(... 'cookiesEncrypted' ...)` — includes encrypted cookies in listings

**10. Proxy passes all Bearer tokens through without validation**
- `src/proxy.js:27-30`: any request with `Authorization: Bearer <anything>` is passed through

**11. Empty directories**
- `worker/src/gatekeeper/` and `worker/src/resume-client/` — empty scaffolding

**12. External ID dedup is weak**
- `worker/src/scraper/linkedin.js:49`, `indeed.js:49`: `externalId = \`${company}-${title}\`.replace(...)` — same company+title in different cities collides

**13. Job detail page linked but doesn't exist**
- `src/app/automation/jobs/page.js:111`: links to `/automation/jobs/${job._id}` but no `[id]/page.js` route exists

**14. `GatekeeperRules` and `JobCriteria` duplicate fields**
- Both have `targetTitles`/`titles`, `minSalary`, work mode booleans, seniority levels
- Unclear which is authoritative for each concern

**15. No rate limiting on API key routes**

**16. `test-pipeline.js` uses simple click, not AI loop**
- Uses `page.$('button[type="submit"], button:has-text("Submit")')` instead of the DeepSeek AI loop
- Should be updated to mirror `apply.processor.js` logic

### Minor Issues

**17. `render.yaml` references `npm run build`** but `package.json` has no build script

**18. `scrapeIntervalMs` in worker config is unused** — scheduler runs on hardcoded `*/30 * * * *` cron

**19. `COOKIE_ENCRYPTION_KEY` env var** checked in both Next.js app (`src/config/env.js`) and worker (`worker/src/config.js`) with different fallback behavior

## Notes
- **MongoDB** is the primary database (not PostgreSQL/Supabase as original spec assumed)
- **Gemini** is the AI provider for resume generation; **DeepSeek** for apply form filling
- **Auth is custom JWT** (no NextAuth.js)
- Puppeteer is installed but unused — it's for the Automation Worker
- All API routes use consistent `x-user-id` from proxy + `requirePermission()` pattern
- To swap any AI task to a different model, edit `src/lib/ai/config.js` or set env var `AI_TASK_<NAME>=provider:model`

## Key Architecture Decisions (Job Automation)

### Database
- **All automation data in MongoDB** — no Supabase/Postgres. Models: `ApiKey`, `JobListing`, `GatekeeperDecision`, `Application`, `PlatformSession`, `JobCriteria`, `SchedulerSettings`, `GatekeeperRules`, `DailyCount`.

### Dashboard Placement
- **Same Next.js app at `/automation/*`** — not a separate deploy. Reuses existing custom JWT auth, MongoDB connection, and deployment.
- **Future split is straightforward**:
  - Copy `/automation/*` pages + API routes → new Next.js app
  - Copy Mongoose models + DB connection utility → both apps
  - Copy auth utilities (`AuthContext`, `requirePermission`, `jwt.js`) → new app
  - Add `MONGODB_URI` + `JWT_SECRET` to new app's env vars (same MongoDB, same JWT secret)
  - CORS headers only needed on Resume Builder routes called by the new Dashboard
  - No auth rewrite — stateless JWT works on both apps with the same secret

### Gatekeeper AI
- **Gemini Flash** via existing `callAI('GATEKEEPER')` in `src/lib/ai/config.js`. Worker calls it through `POST /api/gatekeeper/evaluate` API route (HTTP), not directly.
- **Swappable** — already in `config.js`, change via `AI_TASK_GATEKEEPER=provider:model` env var.

### Apply AI
- **DeepSeek** (`deepseek-chat`) for interactive form filling — analyzes page structure and fills fields with resume data
- **Resume-aware** — fetches master resume, passes it as context to the AI

### Worker → App Authentication
- **API Keys** — `ApiKey` Mongoose model + Bearer token middleware. Future-proof for when they're separate services.

### Resume Modes (future)
- **Mode A**: Custom resume per job — uses job description + profile → `callAI('RESUME_GENERATION')`
- **Mode B**: Same resume for all — uses pre-set profile, no tailoring
- **On-the-fly generation** — no PDFs stored in DB; generate per application

### Templates
- **Exposed via `GET /api/resume/templates`** — returns `{ id, name, ... }` for each template. Worker discovers templates dynamically.
