## How To Run

### Next.js App (port 3000)
```bash
cd src
npm run dev
```

### Worker (port 3001)
Requires Redis running locally.
```bash
cd worker
# .env needs: REDIS_URL, DEEPSEEK_API_KEY, RESUME_BUILDER_URL, RESUME_BUILDER_API_KEY, COOKIE_ENCRYPTION_KEY
node src/index.js
```

### Worker API Key
Create an API key in the dashboard at `/automation/settings/api-keys` or `/api-keys`. Set it as `RESUME_BUILDER_API_KEY` in the worker's `.env`. The worker uses it in `worker/src/db-api.js` to authenticate with the Next.js app via `Authorization: Bearer <key>` header.

## Framework & Tech Stack
- **Framework**: Next.js 16.0.7 (App Router), React 19.2.0
- **Auth**: Custom JWT-based auth (JOSE for Edge, jsonwebtoken for Node) with OTP email login via Brevo
- **Database**: MongoDB via Mongoose
- **AI Providers**: Gemini Flash (resume generation, gatekeeper) + DeepSeek (form filling)
- **PDF**: `@react-pdf/renderer` v4.3.1
- **Styling**: Tailwind CSS v4
- **Worker**: Express + BullMQ + Redis + Playwright

## Auth System
- Custom JWT: access tokens (15min) + refresh tokens (15 days) with rotation
- Proxy (`src/proxy.js`) reads cookies → verifies tokens → injects `x-user-id` header
- Role-based permissions (ADMIN=0, SUBSCRIBER=99, USER=100)
- Credit tracking (Free: 2/day, Pro: 200/month)

## Database (MongoDB)
Models: User, Resume, ResumeMetadata, RefreshToken, Transaction, ApiKey, Application, DailyCount, GatekeeperDecision, GatekeeperRules, JobCriteria, JobListing, PlatformSession, SchedulerSettings, NotificationPrefs, ApplyInstructions

## Pipeline System

### Pipeline Modes (configurable in Scheduler Settings)
| Mode | Behavior |
|------|----------|
| **Scrape Only** | Finds jobs, saves to DB. No gatekeeper evaluation, no applying. |
| **Scrape & Review** | Finds + evaluates jobs. Gatekeeper AI scores each job, decisions saved to DB. Review results manually on job detail pages. |
| **Full Pipeline** | Scrape → gatekeeper → generate tailored resume → DeepSeek AI apply. End-to-end automation. |

The current mode is displayed on the dashboard near the "Scrape Jobs Now" button.

### Worker Pipeline Flow
```
Cron (every 30 min) → scrape.processor
  → Indeed/LinkedIn scrapers (find jobs, get full descriptions)
  → Save to MongoDB (deduped by externalId = company-title-location)
  → Enqueue gate evaluation (only if not scrape_only mode)
    → gate.processor → POST /api/gatekeeper/evaluate (Gemini Flash)
      → AI returns: apply=true/false, confidence, reasoning
      → Decision saved to GatekeeperDecision model
      → Enqueue generate (only if full pipeline mode)
        → generate.processor → POST /api/generate-content
          → Generates tailored resume for this job
        → Enqueue apply
          → apply.processor (DeepSeek AI loop):
            1. Navigate to job page
            2. Check if already applied ("Applied" text detection)
            3. Find & click apply button
            4. Check URL after click:
               - External redirect → save "external_apply"
               - Still on viewjob → re-check for "Applied" text
               - Easy Apply iframe → DeepSeek fills form
            5. AI loop: analyze page → fill fields → click continue → submit
            6. CAPTCHA/blocker detection with user intervention wait
            7. Save application record
```

### Apply Instructions
Users can write plain-English instructions (e.g., "Always use 647-555-0101 for phone", "Set salary to $75,000") at `/automation/settings/apply-instructions`. These are stored in MongoDB via `ApplyInstructions` model, fetched by the worker's `getApplyInstructions()` in `db-api.js`, and injected into the DeepSeek system prompt's RULES section.

### Manual Triggers (from dashboard)
- **Scrape Now**: Button on `/automation` → `POST /api/automation/trigger-scrape` → worker `/trigger/scrape`
- **Apply Now**: Button on job detail page → `POST /api/automation/jobs/[id]/apply` → worker `/trigger/apply-job`

## Configurable Settings (Scheduler)

| Setting | Default | Description |
|---------|---------|-------------|
| enabled | false | Master on/off |
| pipelineMode | scrape_only | scrape_only, scrape_gate, or full |
| dailyRateLimit | 100 | Max API calls per day (used by gatekeeper + generate routes) |
| startHour / endHour | 9 / 18 | Active hours |
| timezone | America/Toronto | |
| activeDays | weekdays | Mon-Fri enabled |
| maxPerDay / maxPerWeek / maxPerRun | 20 / 80 / 5 | Application caps |
| minDelaySeconds / maxDelaySeconds | 60 / 180 | Random delay between applications |
| gatekeeperThreshold | 75 | Min confidence to auto-apply |
| reviewQueueThreshold | 40 | Below this → manual review |
| pauseOnError / pauseOnSessionExpiry | true | Auto-pause triggers |

## API Routes (Automation)

| `/api/automation/daily-count` | GET/POST | Daily usage counter |

## Worker Architecture

### Directory Structure
```
worker/src/
  index.js                ← Express entry point (port 3001)
  config.js               ← Worker config (env vars)
  db-api.js               ← HTTP client to Next.js app API
  redis.js                ← Redis + BullMQ setup
  
  automation/
    browser.js            ← createBrowserContext() — Playwright
    anti-detection.js     ← randomDelay, humanLikeMove
  
  scraper/
    index.js              ← Dispatcher
    indeed.js             ← Indeed scraper (full descriptions via viewjob pages)
    linkedin.js           ← LinkedIn scraper
  
  queue/
    jobs.js               ← BullMQ job definitions
    workers.js            ← Worker pool
    processors/
      scrape.processor.js ← Job scraping → enqueue gate (respects pipelineMode)
      gate.processor.js   ← Gatekeeper eval → enqueue generate (respects pipelineMode)
      generate.processor.js ← Resume generation
      apply.processor.js  ← DeepSeek AI form filling loop
  
  scheduler/
    cron.js               ← Cron (reads settings from MongoDB)
  
  routes/
    health.js             ← /health
    queue.js              ← /queue/*
    trigger.js            ← /trigger/scrape, /trigger/apply, /trigger/apply-job

  notifications/
    index.js              ← Notification helpers

  tests/
    test-apply-with-ai.js ← Working — full AI loop
    test-pipeline.js      ← Updated — calls production applyJobProcessor
```

### Environment Variables (Worker)
- `REDIS_URL` — Redis connection for BullMQ
- `DEEPSEEK_API_KEY` — DeepSeek for form filling
- `RESUME_BUILDER_URL` — Next.js app URL (default: http://localhost:3000)
- `RESUME_BUILDER_API_KEY` — API key created from dashboard
- `COOKIE_ENCRYPTION_KEY` — Must match Next.js app's key
